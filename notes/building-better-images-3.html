Dockerfile Demystified: Best Practices & Tips for Building Better Images

Learn how to craft efficient, maintainable Dockerfiles and leverage the Docker CLI like a pro. This guide covers key instructions, best practices, and tips to help you build smaller, faster, and more secure container images.

üöÄ Intro: What is Docker CLI and Dockerfile

docker (Command Line Interface) is the primary tool for interacting with Docker. It allows you to manage containers, images, networks, and volumes directly from your terminal. With commands like docker build, docker run, and docker ps, you can create, start, stop, and inspect containers, making it an essential part of any Docker workflow.

Dockerfile, on the other hand, is a text file that contains a set of instructions for building a Docker image. It defines how the image is constructed, specifying base images, software dependencies, and configurations. Writing a well-structured Dockerfile is key to creating efficient, reproducible, and maintainable container images.

üöÄ Docker CLI Basics

  • docker help ‚Üí Help on how to use docker
  • docker info ‚Üí Display info. On macOS or Windows, docker info reveals that Docker runs inside a lightweight Linux VM, which acts as the Docker Server behind the scenes.
  • docker version ‚Üí Show the Docker version information (Both Client and Server)
  • docker run ‚Üí Create and run a new container from an image
  • docker exec ‚Üí Lets you run commands inside a running container ‚Äî similar to how ssh allows you to run commands on a virtual machine
  • docker ps ‚Üí Lists all running containers, which are essentially isolated system processes managed by the Docker engine.
  • docker build ‚Üí Build a container image from the instructions defined in a Dockerfile, packaging your application and its environment into a reusable artifact.
  • docker pull ‚Üí Downloads a container image from a registry like Docker Hub, making it available locally for use in your environment.
  • docker push ‚Üí Uploads a locally built image to a container registry, making it accessible for sharing or deployment across different environments.
  • docker images ‚Üí Lists all container images stored locally, showing details like image name, tag, size, and creation time.
  • docker login ‚Üí Authenticates you to a container registry, allowing you to pull or push private images securely.
  • docker logout ‚Üí Ends your session with a container registry, removing stored credentials and securing your environment.
  • docker search ‚Üí Queries Docker Hub for available images, helping you discover official and community-contributed containers.
  • docker inspect ‚Üí Provides detailed, low-level information about Docker objects, such as Os, Architecture, Environment variables, or Labels, in JSON format.
  • docker history ‚Üí Shows the history of an image, listing its layers, creation time, size, and commands used to build it.

üöÄ Anatomy of a Dockerfile

  • FROM ‚Üí Specifies the base image to build upon. Every Dockerfile must start with a FROM.
  • LABEL ‚Üí Adds metadata to your image (e.g., maintainer name, version, description).
  • ARG ‚Üí Defines build-time variables that can be passed during the docker build process using --build-arg.
  • ENV ‚Üí Defines environment variables accessible during build and at runtime.
  • RUN ‚Üí Executes commands inside the image during the build process (e.g., installing packages).
  • COPY / ADD ‚Üí Adds files from your local system into the image.
  • WORKDIR ‚Üí Sets the working directory for subsequent instructions like RUN, CMD, and COPY.
  • USER ‚Üí Specifies the user to run the container as (helps improve security).
  • EXPOSE ‚Üí Documents the port the container will listen on (informational only).
  • ENTRYPOINT ‚Üí Defines the main executable for the container; works with CMD for flexibility.
  • CMD ‚Üí Sets the default command to run when the container starts.

üöÄ Best Practices for Writing Dockerfiles

  • Use Minimal Base Images ‚Üí Start with lightweight glibc-based images (like AL2023 or Debian) for better compatibility and fewer issues.
  • Leverage Multi-Stage Builds ‚Üí Separate build and runtime environments using multi-stage builds. Keeps final images clean and optimized.
  • Order Instructions Efficiently ‚Üí Place frequently changing instructions (like COPY or RUN) near the bottom to maximize layer caching and speed up builds.
  • Use .dockerignore ‚Üí Exclude unnecessary files from the build context to improve performance and reduce image bloat.
  • Avoid Installing Unnecessary Packages ‚Üí Only install what‚Äôs absolutely needed. Clean up temp files or package caches in the same RUN step.
  • Pin Dependency Versions ‚Üí Always specify exact versions for OS packages or tools to ensure reproducible builds.
  • Use ENV for Reusability ‚Üí Store frequently used values (like paths or versions) in ENV variables to simplify updates.
  • Prefer COPY Over ADD ‚Üí Use COPY unless you need ADD features like remote URLs or tar extraction. It‚Äôs more explicit and safer.
  • Label Your Image ‚Üí Add LABEL for metadata like maintainer, version, or description. Helps with image management.
  • Clean Up at the end ‚Üí Remove any temporary files or build artifacts to keep the image lean.
  • Specify a Clear ENTRYPOINT and CMD ‚Üí Define what your container should run and how. Note: ENTRYPOINT runs just before CMD.

üê≥ Real-Life Dockerfile Example: Python Base Image

# syntax=docker/dockerfile:1.5
# Define the base image URL with a default value
ARG IMAGE_URL=amazonlinux:2023
FROM ${IMAGE_URL}

# Maintainer information
LABEL maintainer="you@example.com"

# Define build arguments
ARG PYTHON_VERSION
ARG PYTHON_HOME="/opt/python${PYTHON_VERSION}"

# Set environment variables
ENV PYTHON_VERSION=${PYTHON_VERSION}
ENV PATH="${PYTHON_HOME}/bin:$PATH"

# Install Python and clean up
RUN set -eux \
    # Update system packages
    && dnf update -y \
    \
    # Install Python and development tools
    && dnf install -y python${PYTHON_VERSION} --setopt=install_weak_deps=False \
    \
    # Create a Python virtual environment
    && python${PYTHON_VERSION} -m venv ${PYTHON_HOME} \
    \
    # Upgrade pip in the virtual environment
    && ${PYTHON_HOME}/bin/pip install --no-cache-dir --upgrade pip \
    \
    # Clean up unnecessary files
    && dnf autoremove -y && dnf clean all && rm -rf /var/cache/dnf /var/log/* \
    \
    # Verify the Python installation
    && ${PYTHON_HOME}/bin/python -V \
    && ${PYTHON_HOME}/bin/pip -V

# Set the default command
CMD [ "/bin/bash" ]

üê≥ Real-Life Dockerfile Example: Python App Image

# syntax=docker/dockerfile:1.5
# Define the base image URL with a default value
ARG IMAGE_URL=python3:3.12
FROM ${IMAGE_URL}

# Maintainer information
LABEL maintainer="you@example.com"

# Set the working directory
WORKDIR /app

# Install the application dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy the application code
COPY . .

# Run the application
CMD ["python", "app.py"]

üöÄ Tips & Tricks

  • Pin image versions ‚Üí Avoid using latest (Unless you know how to handle it). Always use specific tags for reproducibility.
  • Clean up cache and temp files ‚Üí Reduce image size by removing package manager caches and logs in the same RUN layer
  • Minimize layers ‚Üí Combine commands using && to keep the number of image layers small.
  • Install only what‚Äôs needed ‚Üí Avoid unnecessary packages‚Äîthis helps reduce image size and attack surface.
  • Avoid secrets in Dockerfile ‚Üí Never hardcode secrets or credentials. Use external secret managers.
  • Check docker [ history | inspect ] ‚Üí Inspect layers, metadata and find size bottlenecks
