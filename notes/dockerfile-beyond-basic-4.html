üöÄ Intro: What is BuildKit?

BuildKit is a powerful and advanced build subsystem introduced by Docker to enhance the way we build images. It‚Äôs an evolution of Docker‚Äôs traditional build process, designed to make builds faster, more efficient, and feature-rich.

Before BuildKit, Docker used a traditional build process where the Docker CLI instructed the Docker daemon to execute each step in the Dockerfile sequentially. This was fine for many use cases, but it lacked flexibility and efficiency.

With the advent of BuildKit, Docker introduced an improved build system capable of parallelizing tasks, optimizing build layers, and handling more complex workflows.

üöÄ Why BuildKit?

  • Faster builds ‚Üí BuildKit supports parallel execution of build steps, leading to faster builds, especially for complex Dockerfiles.
  • Better caching ‚Üí It offers better layer caching strategies, reducing redundant work and speeding up subsequent builds.
  • Multi-platform builds ‚Üí BuildKit allows you to build images for different architectures and operating systems from a single Dockerfile.
  • Advanced features ‚Üí It introduces features like build secrets, advanced caching mechanisms, and more control over the build context.

üöÄ How to Enable BuildKit

To use BuildKit, you simply need to set an environment variable: DOCKER_BUILDKIT=1. This enables the new build system when you run docker build.

üöÄ Multi-Platform Builds with BuildKit

Multi-Platform Images allow you to build and run the same Docker image across different CPU architectures (like amd64, arm64). It ensures your app works on servers, Raspberry Pi, or cloud platforms‚Äîwithout rebuilding for each target.

üê≥ Real-Life Dockerfile Example: Python App Image

üöÄ Intro: What is ONBUILD?

The ONBUILD instruction in Dockerfile is like setting up a trap‚Äîit‚Äôs not triggered when the current image is built, but when another Dockerfile uses this image as its base. Think of it as predefining steps that should run automatically in downstream builds. It‚Äôs especially useful for creating reusable base images that enforce consistent behavior across projects.

  • Trigger Instruction ‚Üí ONBUILD sets up instructions that only run later, when the image is used as a base in another Docker build.
  • Base Image Tool ‚Üí Ideal for base images intended to be extended by others (like language runtimes or frameworks).
  • Delayed Execution ‚Üí Instructions behave as if written just after the FROM in the child Dockerfile.

üê≥ Dockerfile Example: Python Base Image

# syntax=docker/dockerfile:1.5
# Define the base image URL with a default value
ARG IMAGE_URL=amazonlinux:2023
FROM ${IMAGE_URL}

# Maintainer information
LABEL maintainer="you@example.com"

# Define build arguments
ARG PYTHON_VERSION
ARG PYTHON_HOME="/opt/python${PYTHON_VERSION}"

# Set environment variables
ENV PYTHON_VERSION=${PYTHON_VERSION}
ENV PATH="${PYTHON_HOME}/bin:$PATH"

# Install Python and clean up
RUN set -eux \
    # Update system packages
    && dnf update -y \
    \
    # Install Python and development tools
    && dnf install -y python${PYTHON_VERSION} --setopt=install_weak_deps=False \
    \
    # Create a Python virtual environment
    && python${PYTHON_VERSION} -m venv ${PYTHON_HOME} \
    \
    # Upgrade pip in the virtual environment
    && ${PYTHON_HOME}/bin/pip install --no-cache-dir --upgrade pip \
    \
    # Clean up unnecessary files
    && dnf autoremove -y \
    && dnf clean all \
    && rm -rf /var/cache/dnf /var/log/* \
    \
    # Verify the Python installation
    && ${PYTHON_HOME}/bin/python -V \
    && ${PYTHON_HOME}/bin/pip -V

# Define actions that should trigger when another image is built FROM this one
ONBUILD WORKDIR /app
ONBUILD COPY requirements.txt .
ONBUILD RUN pip install --no-cache-dir -r /app/requirements.txt
ONBUILD COPY . .

# Set the default command
CMD [ "/bin/bash" ]
